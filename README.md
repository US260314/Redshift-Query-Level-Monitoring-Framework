## Overview

This repository contains a **serverless monitoring framework for Amazon Redshift Workload Management (WLM)** designed to improve query-level visibility and reduce false-positive alerts generated by traditional node-level CloudWatch monitoring.

Most Redshift monitoring solutions rely on **CloudWatch metrics such as QueriesWaiting and QueueLength**, which are aggregated at the cluster node level. While useful for general system visibility, these metrics often generate **alert noise during transient workload spikes or planned ETL activity**, making it difficult for DBAs to distinguish real performance degradation from normal workload patterns.

This framework introduces a **query-level monitoring approach** by collecting metrics directly from Redshift system tables and applying **stateful breach detection logic** to identify sustained queue pressure.

The solution enables **more accurate alerting, improved operational visibility, and faster incident response for database teams managing Redshift workloads.**

---

# Problem Statement

In large Redshift environments supporting analytics and reporting workloads:

* High concurrency workloads can temporarily increase WLM queue depth.
* CloudWatch node-level metrics frequently generate **false-positive alerts**.
* Alert fatigue makes it harder for DBAs to detect real performance incidents.
* Short-lived spikes are often mistaken for sustained performance degradation.

Traditional monitoring approaches lack:

* Query-level visibility
* Historical breach detection
* Alert suppression for transient spikes
* Business-aware alerting logic

This creates **operational noise and delayed response to real performance issues**.

# Solution Architecture

The framework uses a **serverless architecture built on AWS services** to collect, analyze, and evaluate query-level workload metrics.

### Workflow

1️⃣ **EventBridge Scheduler**

Triggers the monitoring workflow every minute.

---

2️⃣ **Lambda – Redshift Executor**

* Connects to Redshift using the **Redshift Data API**
* Executes a query against internal workload tables

```sql
SELECT
    count(1) as queries_in_queue,
    SUM(CASE WHEN state = 'QueuedWaiting' THEN 1 ELSE 0 END) as queries_waiting,
    SUM(CASE WHEN wait_secs > 300 THEN 1 ELSE 0 END) as queries_waiting_long
FROM admin.v_edw_whats_running_now
```

The query collects:

* Total queries in WLM queues
* Queries currently waiting
* Queries waiting longer than 5 minutes

---

3️⃣ **Metric Storage**

Collected metrics are stored in **S3** with timestamps.

This allows the system to maintain a historical view of workload conditions.

---

4️⃣ **Lambda – Alert History Tracker**

This component evaluates **current metrics against previous samples**.

The framework tracks:

* Breach counters
* Alert state
* Last alert timestamp

Alerts are triggered **only when thresholds are exceeded for multiple consecutive intervals**, preventing alert noise from short spikes.

---

5️⃣ **Alerting**

When sustained thresholds are breached:

* Alerts are published to **SNS**
* On-call teams can receive notifications through **incident management systems (PagerDuty, xMatters, etc.)**

---

6️⃣ **Observability**

Custom **CloudWatch metrics** are published for:

* Queries waiting
* Queries running
* Long-waiting queries
* Breach counters

These metrics can be used to build **Redshift workload dashboards**.

# Example Monitoring Query

The framework relies on a monitoring view built on Redshift WLM system tables.

```sql
SELECT
    convert_timezone('UTC','US/Eastern',swqs.wlm_start_time) AS start_time,
    swqs.state,
    (swqs.queue_time / 1000000.00) AS wait_secs,
    (swqs.exec_time / 1000000.00) AS exec_secs,
    pu.usename,
    cc.name AS service_class_name
FROM stv_wlm_query_state swqs
LEFT JOIN stv_inflight si
    ON swqs.query = si.query
LEFT JOIN pg_user pu
    ON si.userid = pu.usesysid
LEFT JOIN stv_wlm_service_class_config cc
    ON swqs.service_class = cc.service_class;
```

This enables deeper insight into **active query behavior and queue pressure**.
